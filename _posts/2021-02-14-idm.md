## Databases

* Relational databases amount for a **30 billion per year** industry
* Databases store and manage **almost all transactions in the world**
* **Simplify** the life of programmers
* Relational databases are based on **set theory**, more specifically on "relational algebra" and "relational calculus"
* **SQL** is a simple **high-level language** for querying data in wich the user **declares** the final output, but does **not define** how to get it.
  * Things like **query optimizers** make sure to deliver the declared intended result in the most efficent way
* Database Management System (**DBMS**) matains the databases
  * Defines data (type, normalization, constrains) and structure (tables and relations)
  * Physical construction (in storage)
  * Data manipulation (SCRUD)
  * Sharing/protecting (views,user rights)
  * persistence (ACID transaction (all-in or nothing))/recovery (states are reversible)

## Data Modelling (schemas)
* From ER schemas (i.e. chen's notation) to relational logical schemas (tables with foreign keys)
* **Which data is included and how it is structured**.

* 3 phases:
  * **Conceptual design** (UML, chen, EER...)
    * Which entites should be stored? what are their properties? how are they related?
  * **Logical design** (tables, columns, FK, PK...)
    * Which relations should exist? Which attributes do they have? what are the domains (data types) of the attributes, and their constraints?
  * **Physical design** (tablespaces, indexes...)
    * Where and how to store relations, what to index, what metadata to collect?

## Relational Model

* Based on relation**s**
  * Set of tuples i.e.

  %%MyFamSiblings\{(Alex,Sergio),(Alex,Tamara),(Sergio,Alex),\\ (Sergio,Tamara),(Tamara,Alex),(Tamara,Sergio)\}%%

  * The **logical schema** of the secion above specifies the **domain**, in this case could be "people".
  * The **constraint** would be that among those people, their mother is also my mother. 
  * A single relation i.e. (Alex,Sergio) is a subset of the whole set of MyFamSiblings.
    * In this case the cartesian product of Alex,Sergio,Tamara would produce MyFamSiblings
    * In some occasaions the cartersian product does not reflect the real relation (something to be aware of when making queries, as the cartesian product is often an inbetween step for easy compatibility, but things can go wrong, i.e. relations that do not exist may show up)
* A **relational database schema** consists of:
  * set of relation schemas (such as siblings, marriages)
  * Set of integrity constraints (such as Siblings shall not be married)
* A **relational dabasse instance** or state is a "snapshot" of the database (where all constraints are respected)
* **Data Definition Language (DDL)** defines the relation schemas and integrity constraints.
* **Data Manipulation Language (DML)** handles and manipulate the tuples (i.e. actual records in a database table (relation))
* **SQL is used for both DDL and DML**

### Keys

* Primary key (pk): uniquely identifier for a record in a table
* Foreign key (fk): attribute that can be link to a foreign table's primary key (often used for queries to merge data from different tables)
* candidate key: irreducible set of attribute(s) that uniquely identify a tuple.

## Normalization

### 1NF

* No composite fields (i.e. address has to be split into separate fields: postcode, street, city, house number...)
* No lists (i.e. instead of "calculus, algebra, java, web" make a new relation with a table that links students with courses enrolled)

### 2NF

* All columns in a table must depend on the **entire key**
  * Sometimes the entire key is a **composite key** (more than 1 field) and some attributes shown in the table are not really relevant to the whole key, but to a subset. Those attributes are already dependent on another attribute, so they can be safely refractored to another table.

### 3NF

* Attributes depend **directly** on the **entire key**. 3NF has to be 2NF and 1NF, (same holds for 2NF that needs to also be 1NF). The added constraint is that eventhough some attributes do indeed dependend on the entire key, they do it just because a foreign key is dependent on the primary key. For example, a course can have a course id as primary key, name, credits, semester, teacher id, *teacher name...*. Yes, the teacher name depends on the course id, but it can be refractored to a "teacher table" since teacher id is already linked to the course id.
* This is also known as **"no transitivity"**

### Example of EXISTS clause

* Use IN clause when you want to check if it matches a self provided, comma separated, between parenetheses, list of values or a query with a **single** column
* If your logic needs more than 1 column, you use an EXISTS clause where the nested query can use values from the parent query for WHERE clause logic of the nested query

```sql
SELECT COUNT(*) FROM person_100k p WHERE EXISTS
(
SELECT p2.id, p2.name FROM person_100k p2
WHERE p2.id <> p.id AND p.name = p2.name
)
```
## Transactions

* **Atomic** multi-user transactions
* Contain multiple changes that must all be commited or aborted as a single unit and follows ACID principles:
  * Atomicity
    * A transaction is either executed or not
  * Consistency
    * A transaction must result in a legal database state
  * Isolation
    * Other transactions running on parallel do not affect it
  * Durability
    * A successful transaction has permanent effect on the database

## High-level architecture

* The **storage manager** is responsible for
  * storage access
  * file organization
  * indexing and hashing
* The **query processor** parses, optimizes and evalutes queries.
  * Alternatives ways for the query
  * Selection based on cost
    * Which depends on statistical information about relations mantained by the DBMS
* The **transaction manager**
  * Ensures a correct database state despite system failures
  * Controls interactions between transactions to ensure database consistency

## NoSQL and NewSQL

* New tools for new purposes with pros and pros
  * Con: not as fast as relational databases (for relational queries)
  * Pro: can do other stuff well
* Tradeoffs:
  * Transactional consistency for scalability
  * Power for scalability
  * Optimization for flexibility
  
* NoSQL is completely sacrifices core features of RDBMS while NewSQL is a compromise in between NoSQL and Relational Dabases

Terminology:
* Environment: Relational Database Management System (RDBMS) that contains relations and database objects, i.e. MySQL
* clusters: rarely implemented, it's a group of a catalogs, which may be used to have full scope of all database objects in the environment.
* Catalogs: databases inside the environment, i.e. coolworkshop
  * users can connect to a single catalog at a time and have access to all database objects in this catalog
* Schemas: group of tables: i.e. food, training.
* Database objects: tables, domains, constraints that belong to a schema, i.e. all the tables and relations of the food schema (food nutrients, diet, picnic prices)
  * a database object (i.e. food nutrients) can belong exclusively to one schema (i.e. food)
  * database objects can be referenced with their **qualified name** (schemaname.objectname i.e. food.diet)
  * Unless I'm combining things from different schemas (i.e. training and food), it's sufficient to just set a **default schema** with:
    * **SET SCHEMA food**
  * Then "diet" implicitly is recognised as food.diet
  * usually after login the default schema is set to be equal to the username, so change the default schema after login.
* **attribute** = column

### Joins
* default join: Cartesian product by just adding another table in the **FROM** clause:
  * **SELECT** * FROM iceskaters, iceskaters AS iceskaters2 \\
    If you have 10 iceskaters, this will return 100 combinations
  * **Should be adjusted in the WHERE clause** (i.e. iceskaters.team_id = iceskaters2.team_id) to avoid non-existing relations, this would return 5 couples (instead of 100).
* **cartesian product**: Formal definition: Let A and B bet two sets, then the cartesian product of A and B is the set expressed as A x B such that it includes all possible combinations of the objects (a,b), sensitive to the order. Example: the cartesian set of 2 dices is 6 x 6 = 36 total possible combinations from rolling 2 dices. Cartesian product of the digits 0-9 x 0-9 is 100 total possible combinations (numbers 00 to 99).

### Creating and Modifying tables (DDL)

* CREATE SCHEMA schemaname AUTHORIZATION sergio
  * If no schema name is provided it will take the username as schema name
  * if no explicit owner is provided it will take the user that run the query as the owner
  * CREATE TABLE name (columnname columndatatype defaultconstraints CONSTRAINT columnname CHECK (custom-boolean-user-defined-constraint), ...) /* The datatypes are:*/
    * INTEGER
    * FLOAT
    * NUMERIC(precision aka numer of digits,number of digits after decimal point)
    * CHAR(length)
    * VARCHAR(maxlength)
    * DATE
    * TIME
    * TIMESTAMP
  * Default constraints:
    * NOT NULL /* Forces to fill in a value */
    * DEFAULT NULL /* already default option, but useful to un-do when it's not */
    * UNIQUE
    * PRIMARY KEY
    * REFERENCES primary-key-of-another-table
  * Optional settings when referenced rows are deleted:
    * ON DELETE
      * NO ACTION: reject deletion
      * SET NULL: delete and set all referencing foreign keys to NULL
      * CASCADE: delete it along with all rows referring to it
    * ON UPDATE
      * NO ACTION: reject modification
      * CASCADE: change values of referencing foreign keys
    * Default is no action neither on delete nor update
  * You can set composed primary keys by instead of adding PRIMARY KEY after a column, just declare from scratch PRIMARY KEY (column1,column2...)
  * Same for FOREIGN KEY (local-column) REFERENCES (foreign-column)
    * A foreign key can refer to its own table (i.e employee id manages employee id from employee table)
* Delete tables with DROP TABLE tablename deletetype (same for columns)
  * RESTRICT: only tables that have no dependencies
  * CASCADE: all dependent tables are deleted along the way
* Alter (modify) tables with ALTER TABLE tablename operation
  * DROP columnname
  * ADD COLUMN ...(defined the same as when creating a table)

## Impedance mismatch
This term refers to the problem when the DB relational model has different data structures than the host language (i.e. java vs sql)

## Object Persistence
* most application data models are object-oriented
* model mapping is hard for object-oriented programming languages
  * object model differs from the relational model
* all relations between models should be known via the UML diagrams of the conceptual design phase.
* while objects in the host language (i.e. java) have "transient" data (that is, their data is only saved while the application is running), it is possible to store the state of an object in an external file or database, and load it the next time the application is executed ("object data persists").
* **object persistence**: Permantly storing the state of an object
  * benefits:
    * it can be exchanged with other applications
    * easy to replicate
    * can be used by multiple applications at once 
  * cons:
    * to make objects persistent in a RBDMS an explicit identity key needs to be generated
    * called **unique identifier (UID)**
    * objects have complex structures hard to capture in a RDBMS (don't expect to save methods, just attributes)
      * objects need to be "disssambled" and stored in multiple tables linked by foreign keys

### JPA vs JDBC
* JDBC is a standard for Database Access
* JPA is a standard for Object Relational Mapping

JDBC is a standard for connecting to a DB directly and running SQL against it - e.g. `SELECT * FROM USERS`, etc. Data sets can be returned which you can handle in your app, and you can do all the usual things like `INSERT`, `DELETE`, run stored procedures, etc. It is one of the underlying technologies behind most Java database access (including JPA providers).

One of the issues with tradional JDBC apps is that you can often have bad code where mapping between data sets and objects conflict.

JPA is a standard for Object Relational Mapping. This is a technology which allows you to map between objects in code and databse tables. This replaces SQL from the developer with for instance some XML setting files to tell the JPA provider which fields on your object map to which fields in the DB.

## JDBC transactions
* Disabled by default
* Use setAutoCommit(boolean switch) to change transactional behavior
  * true: Every statement is executed immediately
  * false: Statement execution is held back until COMMIT is called
* When transactions are enabled, any number of statements is considered as one transaction until it is committedor canceled.
* to commit a transaction use
  * conn.commit()
* may also create save points
  * conn.setSavepoint(String savepointName)
* to roll back use
  * conn.rollback()
  * or: conn.rollback(String savepointName) to return to a given safe point
* Example:
```java
conn.setAutoCommit(false);
PreparedStatement changeNameStmt = conn.prepareStatement(
  "UPDATE hero SET name=? WHERE name=?");
changeNameStmt.setString(1, "Jean Grey-Summers");
changeNameStmt.setString(2, "Jean Grey");
changeNameStmt.executeUpdate();
changeNameStmt.setString(1, "Scott Grey-Summers");
changeNameStmt.setString(2, "Scott Summers");
changeNameStmt.executeUpdate();
conn.commit();
```

## Manual persistence (easier than to use JDBC)
* Make foreign keys within the java objects
* eager loading:loads the whole object structure as soon as the base object is requested
  * good performance if you usually need the whole structure
* lazy loading: loads only the base object when requested, loads remaining parts of the object’s structure when they are needed
  * good performance if you usually only need parts of the structure
* CRUD read operation (lazy):
```java
public static Villain readPersistent(int uid, Connection conn)
throws SQLException {
  Statement stmt = conn.createStatement();
  Villain villain = new Villain();
  ResultSet rs = stmt.executeQuery("SELECT * FROM villains WHERE uid="+ uid);
  if(rs.next()) {
    villain.uid = uid;
    villain.realName = rs.getString("realName");
    villain.aliases= null;
    } else{
      // handle case when uid does not exist}return villain;}
```

## Relational Algebra (RA)

* Origin of SQL
* Fundamental in the query optimization process
* SQL is declarative whereas RA is procedural (it explicitly determines the steps to compute the result)

### Relational Algebra Operations

* Set algebra (binary operators)
  * Set union \\(\cup\\)
  * Set intersection \\(\cap\\)
  * Set difference \\(\setminus\\)
  * Cartexian product \\(\times\\)
    * A Cartesian product followed by a proper selection is aactually a join
* new relational algebra (unarary operators)
  * Selection \\(\sigma\\) (filtering)
  * Projection \\(\pi\\) (displaying the final result)
  * Renaming \\(\rho\\) (aliases)

Example:

|crs_no|title|
|---|---|
|100|Linear Algebra|
|101|Calculus|
|102|Databases|

This is a relation \\(Course\\{(100,Linear Algebra),(101,Calclus),(102,Databases)\\}\\)

and if we apply selection operator:

%%\sigma_{crs\_no=100}Course = \{(100,Linear Algebra)\}%%

In colloquial terms (a table):

|crs_no|title|
|---|---|
|100|Linear Algebra|

#### Selection

* Selects all tuples (rows) from a relation that satisfy a boolean predicate (in SQL 'WHERE' condition), in RA \\(\sigma_{boolean\ predicate}Relation\\)
* The boolean predicate can be a connection of clauses linked with logical operators
  * \\(\land \lor \lnot\\) (AND, OR, NOT)
* The clauses have the form
  * attribute \\(\theta\\) value
  * attribute \\(\theta\\) attribute
  * value \\(\theta\\) value (i.e. 1=1... useful for debugging)
  * \\(\theta\\) stands for the binary operators \\(= \lt \le \ge \gt \neq\\)

#### Projection

* It is a unary operator on a relation, where it changes all the tupples. It may remove the nth entry of all tuples. In SQL it may also add an extra column (such as a value from a formula or a constant) and rearrange the order of the columns.
* This is the SQL equivalent of only displaying a subset of columns.
* In RA it has the property that duplicate tuples are removed from the relation (a set in set theory does not have duplicates), in SQL duplicates are retained unless "DISTINCT" is used.
* Notation: \\(\pi_{attributeList}Relation\\)

#### Rename

* Useful when applying binary operations from 2 different sets where we need to distinguish the name of the attributes to avoid conflicts.
  * Rename relation: \\(\rho_{s}(Student)\\)
    * SELECT s.* FROM Student AS s
  * Rename attributes: \\(\rho_{(fn,ln)}(\pi_{firstname,lastname}Student)\\)
    * SELECT firstname AS fn, lastname AS ln FROM Student
  * Combination: \\(\rho_{s(fn,ln)}(\pi_{firstname,lastname}Student)\\)
    * SELECT s.firstname AS fn, s.lastname AS ln FROM Student AS s

#### Combination

* You can combine a selection with a projection: 

%%\sigma_{crs\_no=100}\pi_{title}Course = \{(Linear Algebra)\}%%

|title|
|---|
|Linear Algebra|

#### Set algebra operators
* Work like in set theory
* Operands have to be union-compatible
  * Same number of attribues
  * Attributes must have same domain (doesnt matter if they actually refer to the same column name, a string is a string)


### RA Joins
* theta join Defined by \\(R\bowtie_{condition}S\\) or \\(\sigma_{condition}(R\times S)\\) = SQL INNER JOIN
  * Equi-join: Joins only using equality conditions
  * Natural join \\(R\bowtie S\\): cross product that enforce **equality on all attributes with the same name** (therefore it is not necessary to specify the condition as it is implicitly known). It also automatically removes a copy of the joining columns (since they have the same value and therefore redundant)
  * Other theta join could be joins with a condition that is greater than...

### RA Aggregation

* Used in simple (descriptive) statistical computations
  * Merges tuples into groups
  * Computes simple statistics for each group
  * Duplicates are not eliminated
* Creates a new relation, with notation:

%%_{groupingAttributes}\gamma_{functionList}R%%

%%_{course}\gamma_{average(result)}Exam%%

* Sum(attribute)
  * Sum of all non-NULL values
* Average(attribute)
  * average of all non-NULL values
* Maximum(attribute)
  * max of attribute
* Minimum(attribute)
  * min of attribute
* Count(attribute)
  * count attributes not having null values
* Count(*)
  * number of tuples

## Query Processing

* A query is usually stated in a high-level declarative language such as SQL
  * In relational databases it can be mapped to relational algebra
* To be evaluated it's translated into a low-level execution plan, such as "physical relational algebra", which extends relational algebra with primitives to search through internal data structures

### Parser and Translator
* The parser checks for the query syntax and verifies relations, attributes, data types, etc. and checks for typos and basic validations
* The translator will eventually transform the query to a "relationally complete" language with primitives:
  * Tuple scan operators
  * Tuple selection operators
  * Index scan operators
  * Join operators
  * Sort operators
  * Duplicate elimination operator
* Relationally complete are query languages equivalent in expressive power to RA.
* Codd's Theorem
  * Establishes equivalence between Relational Algebra and Tuple/Domain Calculus
    * RA is a variable-free language
    * Tuple/Domain Calculus (equivalent to first-order logic) is a logical language with variables and quantification

### Query execution
* The query optimization determines the specific order of the relational algebra operators
* There are multiple algorithms to evaluate a RA operation
* The evaluation plan is an expression that specifies a detailed evaluation strategy
* The cost of each operation are measured in real time and with an abstract cost given by a) the number of disk accesses, b) CPU time and c) Network communication
* **Usually disk accesses are the predominant factor**, measured by
  * Number of seeks * average seek costs
  * Number of block reads * average block read costs
  * Number of blocks writes * average block write costs
    * Costs for writing a block are higher than costs for reading it, because data is usually also read after writing for verification
* Since CPU time is mostly negligible, it is often ignored
  * Not for in-memory-databases

### Query Optimization

* Transaltion of a query into relational algebra expression leads to a first naive query plan
* They query optimizer transforms it into an efficient plan and handed hover to the evalution engine
* All evaluation plans are usually tree shaped sequences of the relational algebra operators
  * Relations as leaf nodes
  * Operators as internal nodes with one or more childen
* Algebra operators are eventually replaced by algorithmic operators

## Storage media

* DBMS needs to retrieve, update and process persistently stored data
* Data is stored on a storage media, different types differ on:
  * Random Access Speed (time/latency)
    * Measured in ms or ns
    * In some media it is constant, in other such as hard disks speed can very depending on where the data is stored.
  * Random/Sequential Read/Write speed (aka transfer rate)
    * Measured in KB/sec, MB/sec, GB/sec, etc.
    * From fastest to slowest: Register (1-10ns), Cache (10-100ns), Main Memory (60-300ns), Hard Disk (10-12ms), Tertiary Storage (sec-min)
  * Capacity: The fastest the less memory capacity (KB, MB, GB, TB, TB respectively) 
  * Cost per capacity: The fastest the most expensive

### Media characteristics
* Volatile: Memory needs constant power to keep data
  * Dynamic: Gets pumped to keep the data
  * Static: Doesn't need to get pumped periodically
* Access modes:
  * Random: Any randomly stored piece of data has the same access time
  * Sequential: Data con only be accessed in sequential order (thus the closer the fastest)
* Write mode:
  * Mutable storage: Can be read and written arbitrarily
  * Write Once Read Many: WORM

### DBMS storage hierarchy
DBMS will optimize the data distribution among storage levels:
* Primary storage: Fast, limited capacity, high price, usually volatile electronic storage (registers and primary cache)
  * Frequently used data/current work data
* Secondary storage: Slower, large capacity, lower price
  * Main stored data
* Tertiary Storage: Much slower, huge capacity, much lower price, usually offline
  * Backup and long term storage of not frequently used data

### Magentic Disk Storage (HDDs)

* Hard disc drives are still the standard for large, cheap, persistent storage.
  * Used as the main storage media for most data in a database.
* DBMS need to be optimized for efficient disk storage and access
* Data access needs to be as fast as possible
  - Often used data should be accessible with highest speed, rarely needed data may take longer
  - Different data items needed for certain reoccurring tasks should also be stored/accessed together
* Data is divided into sectors (and sectors are grouped into clusters)
* To read or write data the "R/W-head" (i.e. the stylus of a vinil player) must be positioned over the data.
* Latency: time between request to read a block and a rrival of data block in main memory
  * Communication time between processor and disk controller
    * Fraction of a millisecond (that can be ignored)
  * Seektime to position R/W-head at the right cylinder
    * Between 0-42 ms (proportional to the distance that it has to travel, hence the varying speeds)
    * Startime time (1ms). movement time (10-40ms), Stop time (1ms)
  * Rotational delay until first sector of the block appears under R/W-head
    * average 5ms
    * optimization by track cache in disk controller possible
  * Transfer time during disk rotation until all sectors and gaps of the block have passed the R/W-head
    * Around 16 KB-block in 0.5 ms
* The disk controller organizes low level access to the disk
  * head positioning, error checking, signal processing
  * Usally integrated into the disk
  * Provides unified and abstracted interface to access the disks
  * Connects disks to a peripheral bus (i.e. STATA, IDE, FiberChannel, SAS)
* The host bus adapter bridges between the peripheral bus and systems internal bus
  * Internal bus usually integrated into systems main board
* DAS: Directly Attached Storage

### Hard Disk Buffering
* Main memory buffers parts of files
  * When opening a file it will buffer the first parts only
* DBMS does the same in blocks
* Time for reading or writing a block: 10-30 ms
  * During this time, CPU can execute many million instructions
  - I/O-time dominates overall cost
  - Ideally: Block should already be in main memory!

### Controller
* Hardware address of a block is combination of
  - Cylinder number, surface number, block number within track
  - Controller maps hardware address to logical block address (LBA)
* Disk controller transfers content of whole blocks to buffer
  -  Buffer resides in primary storage and can be accessed efficiently
     * Buffer is like a normal cache (but with DB-specific strategies)
* Locating data on a disk is a major bottleneck
* Take-Away-Message HDD
  - Cheap and Large Secondary Storage Medium
  - Avoid Random Accesses!!
    * Reposition head for each access
    * Seek Time + Rotational Delay + Block Transfer
  - Aim for sequential access!
    * Position head once, and then just transfer
  - Aim for re-using data already stored in the Buffer
  - Disks are mechanical devices, and prone to mechanical failure

### IO Model of Computation
* The cost of transferring a block from disk to main memory is orders of magnitude
larger than the cost of the operations in main memory
* Number of (read/write) block accesses is a good approximation of overall cost and should be minimized

#### Avoid Random IO
* Write data on disk in the order of reading
* Use indexes for quickly finding data on disk

### HDD Reliabilty
* Hard drives are not reliable!
 - Drives do fail
 - Means for physical failure recovery are necessary
    * Backups
    * Redundancy
* Hard drives age and wear down. Wear significantly increases by:
 - Contact cycles (head parking)
 - Spindle start-stop
 - Power-on hours
 - Operation outside ideal environment
    * Temperature too low/high
    * Unstable voltage

* Reliability measures are statistical values assuming certain usage patterns
   * Desktop usage (all per year): 2,400 hours, 10,000 motor start/stops,
25°C temperature
   * Server usage (all per year): 8,760 hours, 250 motor start/stops, 40°C
temperature
* Non-Recoverable read errors: A sector on the surface cannot
be read anymore * the data is lost
   * Desktop disk: 1 per 1014 read bits, Server: 1 per 1015 read bits
   * Disk can detect this!
* Maximum contact cycles: Maximum number of allowed head contacts (parking)
   * Usually around 50,000 cycles
* Mean Time Between Failure (MTBF): Statistically
anticipated time for a large disk population failing to 50%
   *  Drive manufactures usually use optimistic simulations to guess
the MTBF
   *  Desktop: 0.7 million hours (80 years), Server: 1.2 million hours
(137 years) – Manufacturers values
* Annualized Failure Rate (AFR): Probability of a failure per
year based on MTBF
   *  AFR = OperatingHoursPerYear / MTBFhours
   *  Desktop: 0.34%, Server: 0.73%

## Access paths
* The traditional problem is that there is a strong performance gap between primary and secondary storage, and that our primary storage is volatile.
* Databases store persistent data, and we can use access paths to get your data qucickly.
* For persistent storage databases are mapped into a number of files.
  * Located in specially protected parts of the file system (tablespaces, etc.)
  * Actually maintained by the operating system
* Efficency depends on:
  * Hardware
  * Phyisical tuning
    * Allocation of data on the disks
      * Each file is partitioned into fixed length blocks (pages)
        * Headercontains general block information like block address, type of segment (data, index,...)
        * Table directory shows whichtables that have rows in the block
        * Row directory contains information about the actual rows in the block (IDs, addresses,...)
        * Row data is actual row data
        * Non-standard blocks are necessary for binary large objects (blobs) and character large objects (clobs)
          * large objects are not interpreted in databases
          * (large) text documents, images, audio and video data
          * Need to be stored in a contiguous sequence of bytes
          * If an object is bigger than a block, contiguous pages of the buffer pool have to be allocated for storage
          * Sometimes preferable to disallow direct access, but only allow access through file-system-like API to allow for fragmentation  
      * Smallest unit transferred from/to storage
      * Block size is specified on DB creation
      * Is a multiple of the OS block size
      * A block usually contains several data records
      * Harddisk Sectors are abstracted by the file system to blocks
      * DBMS abstracts FS blocks to DBMS blocks
    * Intelligent buffer management
      * data access always complete blocks (pages) are transferred from disk into the main memory
      * If the block is buffered, return main memory address
      * Else allocatespace in the buffer, fetchthe block from disk, and return main memory address (like a cache)
      * Writing technique is often "write-back", so only before the cache gets overwritten are the changes written on disk.
        * Least recently used (LRU): the ‘oldest’ block is replaced
        * Toss immediate: After the DB has finished to process a block, the block is immediately replaced
        * Expected Reuse: The block with least expected usefulness is evicted (uses statistics about the frequency of use)
    * Creating access paths and indexing data

### Rows vs columns
Classical approach was rows for the records:
* assumption: we are retreiving the whole row
* Multiple records stored per page
* Easy to add new records
* Efficient when most parts of a records are needed
* Less efficient for large-scale statistics

Columns are the new trend:
* assumption: we are going to access parts of the relations only (i.e. 'price' column)
* Stores column data in different pages
* Efficient compression
* Efficient for column-based statistics
* Column-stores are more I/O efficient for read-only queries as they read, only those attributes which are accessed by a query.
* Re-constructing a whole record is hard 

### Indexing
Just like a dictionary has the letters of the alphabet to easily jump to the pages where your word is.

* Primary Index
  * Created by default for the primary key attributes of a table
  * Index phyisically reorders the whole table
  * Can only have a single primary index, and keep it sorted.
  * Efficent search is possible (i.e. binary search)
  * 
* Secondary Index
  * Optional index for non-primary key attributes
  * Beneficial for speeding up joins on foreign key constraints
  * Builds additional data structure containing the indiex, (usually a binary tree), which costs space for storage and performance for updates.
* There's a trade-off in adding more indexes
  * Space cost & update (insert/delete/update) performance as every index has to be updated, but for read only operations will be faster

```sql
CREATE UNIQUE INDEX indexName ON tableName (columnName DESC);
DROP INDEX indexName;
```

* Primary key columns have an index by default
* For each UNIQUE constraint there is a corresponding index by default
* A binary search on the index file requires fewer block access than a binary search on the entire data file

### Hash-based indexes
* Search keys are distributed uniformly across blocks using a hash function

### Tree-based indexes
* Search keys are stored in sorted order
  * Single-level order indexes
  * Multi-level ordered indexes

### Primary Sparse index
* Primary indexes are often sparse (indexes only a subset of rows, i.e. just the vowels)
* Index record contains pointer to the respetive storage place
* Pros:
  * The number of blocks needed to store the index is small compared to the data
  * Can be kept in buffer
  * Insertion/deletions require few changes in the index
* Cons:
  * Finding non-cached data is costly

### Primary Dense index
* Primary indexes can sometimes be dense (i.e. index all the words of the alphabet)
  * Pros:
    * Wont have the problem of finding non-cached index data
  * Cons"
    * Requires more space than a sparse index and is more costly on insertion/deletion

### Secondary Indexes
* point to locations of records
regarding a non-ordering attribute
* Indexing does not affect storage order
* There can be multiple secondary indexes for the same DB
file
* Secondary indexes are usually dense
* Speeds up retrieval
* Provides logical ordering

### Index sizes
* Primary = number of blocks
* Clustering = number of distinct search keys
* Secondary = up to number of of records in table

### Index structures

* Systems which have very low write load, but high read load \\(\to\\) Bitmap Indexes
* Queries where you only need to check if some records exists \\(\to\\) Bloom filters

## Hash Index
* Supporting Point Queries Eficiently (queries with equality selections)
```sql
SELECT * FROM customers WHERE fname="Sinter" and lname="Klaas"
```

* Mapping of values to an index, therefore O(1) time as we know the exact address. Technically:
* Fixed-Size directly addressable Index Space [0...M] containing M+1 buckets
  * Buckets contain links to data
  * Single link in internal hashing (i.e. in memory)
  * Multiple links for external hashing (i.e. on harddisk)
* Hash Function \\(h:\mathbb{Z} \to [0\dots M]\\)
* Maps any number to [0...M]
* Should be uniform (i.e.: same probability for any x ∈ [0...M])
  * Especially, should also be surjective (i.e. use the full range of [0...M])
* Needs to be deterministic (i.e.: same input → same output)
* Should be simple and fast
  1. Converts a value to a numeric representation (i.e. f(name)=ascii)
  2. Hash the numeric value (we do this to distribute probablity of finding something)
  3. Store block anchor of value in the bucket with computed hash index
* Hash functions are not injective – collisions may happen
* Probability of collision increases with load factor
* If the hashing range [0...M] is exceeded then:
  * Rehashing (start from scratch with a large range)
  * Extendible hashing (uses dynamic directory to double and half number of buckets)
  * Linear hashing (buckets are split into two one after another and are individually rehashed with an additional hash-function)


## Binary Index

### B-Trees
* B-Trees adapt concepts and techniques learned for binary trees
and optimize them for hard disk storage
* Searching within a DB/disk block is very efficient
  * Take advantage of static nature within a block
  * Search can be performed in memory with bisection search
  * Treat entire blocks as tree nodes
* Reading blocks from the disk is expensive
  * Reduce block reads
  * Most data resides in the leaf nodes
* Thus minimize the height of the tree
  * Dramatically increase fan-out factor
  * Tree becomes “bushy”
  * Smaller search path length
* B-Tree are specialized Block Search Trees for disc-based Indexing
* Tree is self-balancing

### B+Trees
* B+Tree is THE super index structure for disk-based databases
* Improved over B-Tree
  * Improved traversal performance
  * Increased search efficency
  * Increased memory efficiency
* If Search key is primary key
  * Dense index: Order of data file is not important
  * Sparse index: data file must be sorted by primary key
* If search key is NOT primary key
  * Search key is not unique
  * Data file is sorted by search key (sparse or dense index)
  * Data file is not sorted by search key (dense index only)
* B trees are nice for queries with inequalities and ranges (<=,<,>>=)
* It uses a recursive algorithm to re-organize the tree after insertion/deletion


## Multidimensional Index

While for 1 attribute hash indexes are best for single equality conditions (b+ do also well), and B+ are best for range conditions, if we introduce more variables then multi-dimensional trees (such as KD tree, R tree) perform better.
